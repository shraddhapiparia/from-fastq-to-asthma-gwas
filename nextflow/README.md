# Nextflow: Downstream Statistical Genetics Workflow

## Objective

Orchestrate the downstream analysis stages — GWAS, polygenic risk score (PRS), and GWAS-to-eQTL annotation — as a single reproducible workflow using Nextflow DSL2.

---

## What This Workflow Does

This workflow wraps three existing modules and connects them through their output files:

```
05_population_pca/results/   (prerequisite — must exist before running)
    ↓
GWAS  (06_asthma_gwas)               asthma_gwas.assoc.logistic
    ↓                 ↓
  PRS               EQTL
(07_polygenic_      (09_gwas_to_eqtl)
 risk_score)
```

- `GWAS` runs first and produces `asthma_gwas.assoc.logistic`
- `PRS` and `EQTL` branch from the GWAS output and run in parallel
- All analysis logic lives in the individual module directories — nothing is reimplemented here

---

## Scope

This workflow covers downstream statistical genetics only:

| Stage | Module | Script called |
|-------|--------|---------------|
| GWAS | `06_asthma_gwas` | `scripts/run_gwas.sh` |
| PRS | `07_polygenic_risk_score` | `scripts/run_prs.sh` |
| eQTL annotation | `09_gwas_to_eqtl` | `scripts/run_eqtl.sh` |

FASTQ QC, alignment, variant calling, and variant annotation are not included in this workflow layer. They can be added upstream in a future iteration (see Future Directions below).

## Prerequisites

**1. Module 05 outputs must exist:**

```bash
cd 05_population_pca
conda activate population_pca_env
bash scripts/run_qc_pca.sh
```

Required files:
- `05_population_pca/results/demo_genotypes_qc.{bed,bim,fam}`
- `05_population_pca/results/demo_genotypes_pca.eigenvec`

**2. Nextflow must be installed:**

```bash
conda install -c bioconda nextflow
# or
curl -s https://get.nextflow.io | bash
```

**3. The `genomics_pipeline` conda environment must be active:**

```bash
conda activate genomics_pipeline
```

This environment provides Python, R, PLINK, and all dependencies used by the module scripts. The workflow does not activate environments internally.

---

## Running

From the **repo root**:

```bash
conda activate genomics_pipeline
nextflow run nextflow/main.nf -params-file nextflow/params.json
```

To override a parameter without editing `params.json`:

```bash
nextflow run nextflow/main.nf \
    -params-file nextflow/params.json \
    --prs_p_threshold 0.05 \
    --eqtl_top_n 10
```

To resume a partially completed run (Nextflow caches completed processes):

```bash
nextflow run nextflow/main.nf -params-file nextflow/params.json -resume
```

---

## Parameters

All paths in `params.json` are relative to the repo root. Nextflow resolves them against `launchDir` (the directory from which `nextflow run` is called).

| Parameter | Default | Description |
|-----------|---------|-------------|
| `geno_prefix` | `05_population_pca/results/demo_genotypes_qc` | PLINK binary prefix for PRS scoring |
| `pheno_file` | `06_asthma_gwas/results/asthma_pheno.txt` | Phenotype file generated by the GWAS step |
| `prs_out_prefix` | `07_polygenic_risk_score/results/asthma_prs` | Output prefix for all PRS files |
| `prs_p_threshold` | `0.5` | P-value threshold for variant selection in PRS |
| `eqtl_ref` | `09_gwas_to_eqtl/data/demo_eqtl_reference.tsv` | eQTL reference table |
| `eqtl_out_prefix` | `09_gwas_to_eqtl/results/eqtl_annotation` | Output prefix for eQTL annotation files |
| `eqtl_top_n` | `20` | Number of top GWAS hits to annotate |

## Implementation Notes

- **`val` channels**: the GWAS output path is passed between processes as a string value, not a staged Nextflow `path`. This avoids unnecessary file copying and keeps outputs in their expected module locations.
- **Working-directory guards**: each module script checks that it is run from its own directory. The Nextflow process blocks handle this with `cd "${launchDir}/..."` before calling `bash scripts/...`.
- **No reimplemented logic**: all biological computation lives in the module scripts. This workflow only sequences their execution and wires the GWAS output path to the PRS and EQTL processes.
- **Caching**: Nextflow caches completed processes by hash. Use `-resume` to skip already-completed steps after a partial failure.
